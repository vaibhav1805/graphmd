Create `internal/code/comments/` package implementing comment-based dependency extraction.

**CommentSyntax enum:**
```go
type CommentSyntax int
const (
    SyntaxGo         CommentSyntax = iota // // and /* */
    SyntaxPython                          // # and """ and '''
    SyntaxJavaScript                      // //, /* */, and /** */
)
```

**Core function `Analyze(lines []string, syntax CommentSyntax, knownComponents map[string]bool) []code.CodeSignal`:**

The function scans lines for comment regions and applies pattern matching to extract dependency signals. It must:

1. **Track comment state** across lines (block comments, docstrings):
   - Go: `/* ... */` blocks
   - Python: `""" ... """` and `''' ... '''` blocks (docstrings)
   - JavaScript: `/* ... */` and `/** ... */` blocks
   - Use the same state-tracking approach as jsparser/parser.go lines 89-129

2. **Detect single-line comments** by syntax:
   - Go/JS: lines starting with `//` (after trimming whitespace)
   - Python: lines starting with `#` (after trimming whitespace)
   - Strip the comment prefix to get the text content

3. **Apply patterns to comment text:**

   a. **Explicit dependency patterns** (case-insensitive):
      `(?i)(?:calls|depends on|uses|connects to|talks to|sends to|reads from|writes to)\s+([\w][\w.-]*)`
      - Extract the component-like name after the verb
      - Confidence: 0.5 if name is in knownComponents, 0.4 otherwise

   b. **TODO/FIXME patterns** with component-like names:
      `(?i)(?:TODO|FIXME|HACK|XXX)[:\s]+.*?([\w][\w.-]*(?:-(?:service|api|db|cache|queue|broker|cluster))[\w.-]*)`
      - Only match names with infrastructure-like suffixes to reduce false positives
      - Confidence: 0.3 (ambiguous/speculative)

   c. **URL references in comments:**
      - Match URLs in comment text using a URL pattern
      - Parse with `net/url` and extract hostname
      - Filter out documentation domains (example.com, localhost, etc. — reuse filtering logic from connstring or inline)
      - Confidence: 0.5 if hostname is in knownComponents, 0.4 otherwise
      - TargetType: infer from URL scheme if possible (same scheme-to-type mapping as connstring)

4. **Build CodeSignal for each match:**
   - SourceFile: left empty (caller sets it)
   - LineNumber: the 1-based line number where the match occurs
   - TargetComponent: the extracted name/hostname
   - TargetType: inferred from context or "unknown"
   - DetectionKind: "comment_hint" for all (per CONTEXT.md)
   - Evidence: the comment text (trimmed, max 200 chars)
   - Language: left empty (caller sets it)
   - Confidence: tiered as specified above

**TDD approach:** Write tests FIRST covering:
- Go single-line comments: `// Calls payment-api`, `// Depends on Redis`
- Python comments: `# Uses primary-db`, `# Connects to cache-cluster`
- JS comments: `// Talks to user-service`
- Block comments (Go `/* */`, JS `/** */`): patterns inside block comments
- Python docstrings: patterns inside `"""..."""` blocks
- TODO/FIXME: `// TODO: migrate to new-db-service`, `# FIXME: redis-cache timeout`
- URL in comments: `// endpoint: http://user-service/api/v1`, `# redis://cache:6379`
- Known component boosting: same pattern gets 0.5 when name is in knownComponents, 0.4 when not
- Filtered URLs: `// see http://example.com` should NOT produce a signal
- Mixed content: file with both regular code and comments — only comments produce signals
- Multi-line docstrings spanning multiple lines
- Edge cases: empty lines, comments with no dependency info, comment-like strings inside code

Then implement to make tests pass.
